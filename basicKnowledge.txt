Metadata Filtering & Parent-Child Chunking
Two powerful RAG techniques that solve very different problems. Let's go deep on each.

1. Metadata Filtering
What it is
Every chunk you store in a vector DB can carry metadata — arbitrary key/value pairs attached to the chunk (e.g. page=12, section="Clustering", source="MachineLearning.pdf", date="2024-03-01", author="Burkov", doc_type="research_paper").

Metadata filtering means: during retrieval, you don't just do similarity search over all chunks — you first narrow the search space to chunks whose metadata matches some condition, then do similarity + ranking on that filtered subset.

Think of it like SQL WHERE + vector search:

SELECT * FROM chunks
WHERE section = 'Clustering' AND page BETWEEN 100 AND 150
ORDER BY vector_similarity(embedding, query_embedding)
LIMIT 5
Why it matters
Pure vector search has a big weakness: it doesn't understand hard constraints. If the user asks:

"What did the 2024 paper say about transformers?"

Vector search will happily return chunks from 2019 papers that mention transformers, because semantically they're very similar. The model has no notion of "the year must be 2024" — that's a hard filter, not a soft similarity.

Metadata filtering fixes this.

How metadata gets attached
When you load and split documents, each chunk gets a metadata dict. In your own pipeline (create_database.py):

PyPDFLoader automatically attaches {"source": "MachineLearning.pdf", "page": <page_number>} to every chunk. You could enrich this yourself:

for chunk in chunks:
    chunk.metadata["book"] = "The Hundred-Page ML Book"
    chunk.metadata["author"] = "Burkov"
    # even derived metadata:
    if "cluster" in chunk.page_content.lower():
        chunk.metadata["topic"] = "clustering"
How filtering works at query time
With Chroma + LangChain:

retriever = vectorStore.as_retriever(
    search_kwargs={
        "k": 5,
        "filter": {"topic": "clustering"}   # <-- metadata filter
    }
)
Chroma supports richer operators too:

"filter": {
    "page": {"$gte": 100, "$lte": 150},
    "topic": {"$in": ["clustering", "classification"]}
}
This filter is applied before similarity scoring — so vector search happens only over matching chunks.

Two flavors of filtering
a) Static / manual filtering You (the developer) hardcode a filter based on app logic. Example: your app has separate tabs for "Research Papers" vs "Internal Docs" — each tab sets its own filter.

b) Self-querying / dynamic filtering The LLM itself extracts the filter from the user's question. LangChain calls this a SelfQueryRetriever. Flow:

User asks: "What did Burkov say about clustering on pages 100–150?"
An LLM call parses this into: filter = {"author": "Burkov", "page": {"$gte": 100, "$lte": 150}} and query = "clustering".
Vector search runs with that filter.
This is powerful but costs you one extra LLM call per query.

2. Parent-Child Chunking (a.k.a. Parent Document Retriever / Small-to-Big Retrieval)
The chunking dilemma
When splitting documents, you face a painful tradeoff:

Small chunks (e.g. 200 tokens): embeddings are precise, retrieval is accurate — but the chunk is too narrow, missing surrounding context the LLM needs to answer well.
Large chunks (e.g. 2000 tokens): lots of context for the LLM — but embeddings are fuzzy (one vector tries to represent many topics), and retrieval accuracy drops.
Your current setup:

1000 tokens is a compromise. You feel the pain when a single chunk talks about both k-means and HDBSCAN — the embedding is a blur of both.

The idea
Keep two representations of the document:

Child chunks: small (e.g. 200–400 tokens). Embedded and stored in the vector DB. Used for searching.
Parent chunks: large (e.g. 1500–2000 tokens, or even whole pages/sections). Stored in a regular key-value store, not embedded. Used for answering.
Each child knows which parent it came from (via a parent ID in metadata).

Flow at query time
User query → embedded.
Vector search finds top-k small child chunks (precise matching).
For each retrieved child, look up its parent.
Deduplicate parents (multiple children may point to the same parent).
Feed the parents into the LLM as context.
You get the best of both: precise retrieval, rich context.

                         comparison with k-means, etc.)
   ↓ LLM answers using Parent A
LangChain implementation sketch
from langchain.retrievers import ParentDocumentRetriever
from langchain.storage import InMemoryStore
from langchain_text_splitters import RecursiveCharacterTextSplitter
parent_splitter = RecursiveCharacterTextSplitter(chunk_size=1800, chunk_overlap=200)
child_splitter  = RecursiveCharacterTextSplitter(chunk_size=300,  chunk_overlap=50)
docstore = InMemoryStore()  # holds parents by id
retriever = ParentDocumentRetriever(
    vectorstore=vectorStore,       # stores children
    docstore=docstore,             # stores parents
    parent_splitter=parent_splitter,
    child_splitter=child_splitter,
)
retriever.add_documents(pdfDocuments)
Internally LangChain:

Splits docs into parents, assigns each a UUID, stores them in docstore.
Splits each parent further into children, attaches {"doc_id": <parent_uuid>} to each child, embeds children into vectorStore.
At query time, searches children, reads their doc_id metadata, fetches parents from docstore, returns parents.
Variants
a) Small-to-big Embed a small window (1–3 sentences), return a larger window (e.g. the whole paragraph). Sometimes also called "sentence window retrieval" (LlamaIndex term).

b) Summary-to-full Embed an LLM-generated summary of each section, but on retrieval return the full section. Great when sections are long and varied — the summary is a clean semantic fingerprint.

c) Hypothetical question-to-chunk (HyDE-ish) For each chunk, an LLM generates 3 questions that chunk could answer. Embed those questions. At query time, user question matches one of the generated questions → retrieve the original chunk. Brilliant for FAQ-style content.

Why it helps in your case
Look at what happened in your terminal:

User: "What are different types of clustering?" AI: lists DBSCAN, k-means, HDBSCAN, GMM, Spectral, Hierarchical.

The model only listed them briefly because each type was probably in a different 1000-token chunk, and your retriever grabbed a few per type — so each got a shallow mention.

With parent-child chunking, each child (e.g. "DBSCAN eps parameter") would match precisely, but the returned parent would be the entire "Clustering Algorithms" section. You'd get a vastly richer answer — probably enough that the follow-up "explain each in detail" wouldn't even be needed.

How they combine
These techniques are orthogonal — you can and usually should stack them:

Parent-child chunking → better context grounding.
Metadata filtering → narrow to the right document / section / date.
Hybrid search (BM25 + vector) → you already do this.
History-aware query rewriting → the thing we discussed earlier.
A production-grade RAG pipeline typically has all four. The combined effect is much larger than any one of them alone.

If I were upgrading your project in priority order
Add parent-child chunking (biggest answer-quality jump for textbook PDFs).
Add metadata filtering by source (so when you add more PDFs, they don't contaminate each other).
Keep hybrid + add history rewriting (already discussed).
Optionally add a reranker (Cohere rerank, bge-reranker) as a final pass — this is the single cheapest 10–20% quality win on top of everything else.
